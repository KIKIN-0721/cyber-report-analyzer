## 1. 设计目标

证据映射模块（`src/evidence/`）的核心目标是：让每一条由 OCR 或正文提取产生的识别结果，都能被**唯一、稳定、可回放**地追溯到原始文档中的具体位置。

本文档冻结 S1/S2 阶段证据映射的数据结构、生成规则与接口约定，作为 `src/evidence/tracer.py` 及下游报告导出模块的开发基线。

## 2. 核心数据结构

### 2.1 EvidenceTraceV3（单条证据追溯记录）

对齐 `docs/architecture/stage-interfaces/S3-接口规范.md` 中定义的 `EvidenceTraceV3`，S1 提前固化以下字段：

```json
{
  "evidence_id": "EVI-task-0001-003-02",
  "rule_id": "S1-RSA-001",
  "field": "crypto.rsa.key_length",
  "value": "2048",
  "page": 3,
  "snippet": "RSA2048",
  "image_ref": "img-0001",
  "bbox": {"x": 120, "y": 340, "w": 400, "h": 80},
  "source_type": "image_ocr",
  "confidence": 0.91,
  "correction_type": "none",
  "created_at": "2026-04-15T12:00:00Z"
}
```

**字段语义表**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `evidence_id` | string | 是 | 全局唯一证据标识，生成规则见第 3 节 |
| `rule_id` | string | 否 | 关联的规则 ID（S3 规则引擎执行后回填） |
| `field` | string | 是 | 标准字段名，如 `crypto.rsa.key_length` |
| `value` | string | 是 | 提取归一化后的值，如 `2048` |
| `page` | integer | 是 | 该证据在 PDF 中的原始页码，从 1 开始计数 |
| `snippet` | string | 是 | 原始文本片段，建议截取命中词前后各 20 字符 |
| `image_ref` | string | 否 | 关联的图片 ID，对应 `OCRResultV2.image_id` 或 `ParseResultV2.images[].image_id` |
| `bbox` | object | 否 | 命中区域在图片内的相对坐标，格式见第 4 节 |
| `source_type` | string | 是 | 来源类型：`text`（正文提取） / `image_ocr`（截图识别） / `table`（表格提取） |
| `confidence` | number | 是 | 提取置信度；`text` 类型可统一设为 `1.0` |
| `correction_type` | string | 否 | 纠错类型：`none` / `dict` / `regex` |
| `created_at` | string | 否 | ISO 8601 时间戳 |

### 2.2 OCR 中间态证据对象（OCREvidence）

在 OCR 模块内部，从 `OCRResultV2` 向 `EvidenceTraceV3` 转换前，使用以下中间结构：

```json
{
  "image_id": "img-0001",
  "page": 3,
  "field": "crypto.rsa.key_length",
  "value": "2048",
  "raw_text": "RSA2048",
  "bbox": {"x": 120, "y": 340, "w": 400, "h": 80},
  "confidence": 0.91,
  "correction_type": "none"
}
```

**说明**：该结构由 `src/ocr/post_processor.py` 在字段提取阶段生成，随后传入 `src/evidence/tracer.py` 生成最终的 `EvidenceTraceV3`。

### 2.3 正文提取证据对象（TextEvidence）

由 parser 模块产生的正文字段，同样需映射为证据：

```json
{
  "page": 2,
  "field": "crypto.tls.version",
  "value": "1.2",
  "raw_text": "The device supports TLS 1.2 and above.",
  "bbox": null,
  "confidence": 1.0,
  "correction_type": "none"
}
```

**说明**：
- `source_type` 为 `text`
- `image_ref` 为空（或省略）
- `bbox` 为空（或省略），因为正文提取目前不精确到字符坐标

## 3. evidence_id 生成规则

`evidence_id` 是证据溯源的锚点，必须在**任务级别**全局唯一，且在任务生命周期内稳定不变。

### 3.1 生成公式

```
evidence_id = "EVI-{task_id}-{page:03d}-{seq:02d}"
```

**各段含义**：
- `EVI`：固定前缀，标识该 ID 为 Evidence
- `task_id`：来自 `web.submit_report()` 生成的任务 ID，如 `task-0001`
- `page`：PDF 原始页码，三位零填充（`001` ~ `999`）
- `seq`：该页内第几条证据，两位零填充（`01` ~ `99`）

**示例**：
- 第 1 页第 1 条：`EVI-task-0001-001-01`
- 第 3 页第 2 条：`EVI-task-0001-003-02`

### 3.2 生成时机与计数器管理

- **生成时机**：在字段提取完成、即将输出 `StructuredField` 或 `EvidenceTraceV3` 时生成
- **计数器范围**：以 `task_id` + `page` 为作用域，每条新证据 `seq` 自增 1
- **持久化要求**：S2 阶段可在 `InMemoryRepository` 中维护 `(task_id, page) -> seq` 的计数器；S3/S4 迁移至数据库表时，建议用原子自增或 UUID 后缀避免并发冲突
- **稳定性要求**：同一份 PDF 在同一任务内重复解析，必须生成相同的 `evidence_id`（即计数器只与页内命中顺序有关，与运行时间无关）

### 3.3 扩展规则（S4 及以后）

若单页证据超过 99 条，扩展为三位序列号：
```
EVI-task-0001-003-100
```
S1/S2 阶段暂不要求处理该边界，但代码实现时建议不做长度硬限制。

## 4. bbox 坐标系约定

### 4.1 坐标定义

`bbox` 采用**图片左上角原点**的相对坐标系：

```json
{
  "x": 120,
  "y": 340,
  "w": 400,
  "h": 80
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `x` | float | 命中区域左上角距离图片左边缘的像素值 |
| `y` | float | 命中区域左上角距离图片上边缘的像素值 |
| `w` | float | 命中区域宽度（像素） |
| `h` | float | 命中区域高度（像素） |

### 4.2 数据来源

- **OCR 来源**：直接复用 PaddleOCR 返回的 `bbox` 信息。PaddleOCR 返回的是四点坐标 `[[x1,y1],[x2,y2],[x3,y3],[x4,y4]]`，转换为矩形 `bbox` 的公式为：
  ```python
  x = min(p[0] for p in bbox)
  y = min(p[1] for p in bbox)
  w = max(p[0] for p in bbox) - x
  h = max(p[1] for p in bbox) - y
  ```
- **正文来源**：当前 parser（`src/parser/pdf_parser.py`）尚未输出字符级坐标，正文提取的 `bbox` 暂为 `null`。S3 若 parser 引入 `pdfplumber` 等工具支持字符坐标，再补充填充。

### 4.3 高亮图生成（S3/S4）

证据导出模块（`src/evidence/exporter.py`，S3 实现）在生成高亮图时：
1. 读取 `image_ref` 对应的原始图片文件
2. 在 `bbox` 位置绘制红色矩形框（推荐线宽 2px，颜色 `#FF0000`）
3. 输出高亮后图片，文件名建议：`EVI-task-0001-003-02_highlight.png`

## 5. snippet 截取规则

`snippet` 用于快速展示证据上下文，应同时包含命中词本身和少量前后文。

### 5.1 截取逻辑

```python
def extract_snippet(full_text: str, hit_word: str, window: int = 20) -> str:
    idx = full_text.find(hit_word)
    if idx == -1:
        return full_text[:window * 2]
    start = max(0, idx - window)
    end = min(len(full_text), idx + len(hit_word) + window)
    return full_text[start:end]
```

**规则说明**：
- `window` 默认值 **20 个字符**（中文按字符计，英文同样按字符计）
- 若命中词在 `full_text` 中未找到（如经正则归一化后原文差异较大），则截取前 40 个字符
- 输出不添加 `"..."` 省略号，保持简洁；导出展示时由前端或报告模块自行补充

### 5.2 示例

- 原文：`"The device is configured with RSA2048 key length for all TLS sessions."`
- 命中词：`RSA2048`
- snippet：`"configured with RSA2048 key length for"`

## 6. 映射流程（Pipeline）

```text
[Parser / OCR 输出]
        ↓
┌─────────────────────────┐
│  1. 字段提取与归一化     │  ← src/ocr/post_processor.py
│     输出 OCREvidence     │
└──────────┬──────────────┘
           ↓
┌─────────────────────────┐
│  2. evidence_id 生成     │  ← src/evidence/tracer.py
│     按 task_id + page    │
│     分配全局唯一 ID      │
└──────────┬──────────────┘
           ↓
┌─────────────────────────┐
│  3. 证据追溯记录组装      │  ← src/evidence/tracer.py
│     输出 EvidenceTraceV3 │
└──────────┬──────────────┘
           ↓
┌─────────────────────────┐
│  4. 结构化字段输出        │  ← 与 parser 协同生成 StructuredField
│     进入规则引擎         │
└──────────┬──────────────┘
           ↓
[Rules Engine / Storage / Web]
```

**关键约定**：
- `OCREvidence` → `EvidenceTraceV3` 的转换是**无损**的，所有原始字段必须保留
- `EvidenceTraceV3` 中的 `rule_id` 在 S3 之前可为空字符串，待规则引擎执行后回填
- 同一条 `OCREvidence` 若被多条规则引用（例如 `value=2048` 同时触发不同规则），应生成**多条** `EvidenceTraceV3`，仅 `rule_id` 不同，`evidence_id` 保持不变


### 7.2 S3/S4 阶段：数据库表设计

对接 `docs/architecture/stage-interfaces/` 系列文档中的数据库建议，证据相关记录最终应落入以下两张表：

**表 A：`extracted_fields`**（S2 即需设计）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INT PK | 自增主键 |
| `document_id` / `task_id` | VARCHAR | 关联任务 |
| `field_name` | VARCHAR | 标准字段名 |
| `field_value` | VARCHAR | 归一化值 |
| `normalized_value` | VARCHAR | 与 `field_value` 相同或更精简 |
| `source_type` | VARCHAR | `text` / `image_ocr` / `table` |
| `page_no` | INT | 页码 |
| `snippet` | TEXT | 片段 |
| `confidence` | FLOAT | 置信度 |
| `bbox_info` | JSON | `bbox` 对象序列化 |
| `created_at` | DATETIME | 创建时间 |

**表 B：`rule_results`**（S3 引入）

| 字段 | 类型 | 说明 |
|------|------|------|
| `evidence_ref` | VARCHAR | 关联 `extracted_fields` 中的 `evidence_id` 或自增 `id` |

**建议**：S2 在内存存储时即按上述字段名实践，避免 S3 切数据库时返工。

OCR 模块调用证据模块的接口：

```python

    """
    输入 hit 必须包含的字段：
      - field: str
      - value: str
      - page: int
      - snippet: str
      - image_ref: str（可选）
      - bbox: dict（可选）
      - source_type: str
      - confidence: float
      - correction_type: str（可选）
    """
    ...

def build_evidence_batch(hits: List[Dict[str, str]]) -> List[Dict[str, Any]]:
    ...
```

### 8.2 下游输出（供 Rules Engine / Reporting / Web）

规则引擎消费证据的方式：
- 规则引擎根据 `field` + `value` 做判定
- 判定结果中通过 `evidence_ref` 字段回指 `evidence_id`
- Web 展示时，通过 `evidence_id` 查询 `EvidenceTraceV3`，获取 `page`、`snippet`、`image_ref`、`bbox`

## 9. 风险与边界说明

| 风险 | 影响 | 应对 |
|------|------|------|
| Parser 不输出 `page` | `EvidenceTraceV3.page` 缺失，溯源失败 | S2 第一周与叶泽东强制对齐 `ParseResultV2.images[].page` |
| Parser 不输出 `image_id` | `image_ref` 无法关联 | S2 第一周与叶泽东对齐 `ParseResultV2.images[].image_id` |
| 单页命中超过 99 条 | `evidence_id` seq 溢出 | 代码实现时不限制 seq 位数，用字符串格式化动态适配 |
| bbox 坐标系不一致 | 高亮图位置偏移 | 明确以图片左上角为原点，与 parser 统一坐标系定义 |
